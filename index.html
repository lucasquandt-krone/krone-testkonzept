Kapitel 1 – Bausteine eines Testkonzepts

Testfall

Ein Testfall ist die kleinste dokumentierte Einheit im Testmanagement. Er beschreibt:

→ Was geprüft wird (Testobjekt)
→ Wie geprüft wird (Anweisungen / Instructions)
→ Was erwartet wird (Expected Result)
→ Welche Daten benötigt werden (Testdaten)

Ein Testfall besteht aus einem oder mehreren Testschritten. Jeder Testschritt enthält eine Anweisung und ein erwartetes Ergebnis.

Beispiel: Kundenauftrag mit Standardartikel anlegen
Nr.AnweisungErwartetes Ergebnis
1Transaktion „Kundenaufträge anlegen“ öffnenTransaktion lässt sich öffnen
2Auftragsart „TA“ auswählenAuftragsart wird übernommen
3Kunde eingebenKundendaten werden geladen
4Material und Menge eingebenPreis wird ermittelt
5Auftrag sichernAuftragsnummer wird erzeugt
💡

Qualitätskriterien

Eindeutig – Jeder Schritt ist so beschrieben, dass ein anderer Key User ihn ohne Rückfragen ausführen kann. Messbar – Das erwartete Ergebnis ist konkret und überprüfbar (z.B. „Auftragsnummer wird erzeugt“, nicht „Auftrag funktioniert“). Unabhängig von Testdaten – Die Anweisungen beschreiben das Was, nicht das konkrete Welcher Kunde – Testdaten werden separat gepflegt.

Testplan

Ein Testplan bündelt zusammengehörige Testfälle zu einer logischen Einheit. Er definiert:

📋
Welche Testfälle enthalten sind
📅
Welcher Testphase zugeordnet
👤
Wer die Tests durchführt
📊
Welche Testdaten verwendet

Teststatus und Ergebnisse

Jeder Testfall durchläuft einen Lebenszyklus:

–

Initial

Noch nicht gestartet

→
▶

In Progress

Wird aktuell getestet

→
✓

Passed

Alle Schritte liefern das erwartete Ergebnis

oder
✗

Failed

Mindestens ein Schritt weicht ab → Defect wird angelegt

Defect (Fehler)

Ein Defect ist eine dokumentierte Abweichung zwischen erwartetem und tatsächlichem Ergebnis. Er enthält:

📝
Beschreibung des Fehlers
🔄
Reproduktionsschritte
⚡
Priorität (Very High, High / Medium / Low)
🔗
Automatische Zuordnung zum Testfall und Testschritt
📸
Evidenz (Screenshot, Belegnummern)

Testdaten

Testdaten sind die konkreten Eingabewerte, die für die Testdurchführung benötigt werden. Sie müssen vor der Testphase definiert und im Testsystem angelegt sein.

📦 Stammdaten
Materialstämme, Kundenstämme, Lieferanten, Kontenplan
📄 Bewegungsdaten
Bestehende Aufträge, Bestellungen, Belege
⚙️ Konfigurationsdaten
Buchungskreise, Werke, Lagerorte
Kapitel 2 – Rollen im Testmanagement
🎯
Steuerung

Testmanagement-Leitung (PMO)

→ Gesamtverantwortung für die Planung und Steuerung aller Testaktivitäten
→ Einrichtung und Administration des Test-Tools (SAP Cloud ALM)
→ Reporting an Lenkungsausschuss und Projektleitung
→ Eskalationsinstanz bei Blockern
⭐
Key User

Key User

→ Verantwortlich für die Testfälle seines Fachbereichs/Moduls
→ Koordiniert die Tester und verteilt die Arbeitspakete
→ Qualifiziert Defects: Ist es ein IT-Problem oder ein fachliches Missverständnis?
→ Ansprechpartner zwischen Fachbereich und IT
👤
Testdurchführung

Key User (Tester)

→ Führt die Testfälle im System durch
→ Dokumentiert das tatsächliche Ergebnis in SAP Cloud ALM
→ Meldet Defects mit ausreichender Beschreibung und Evidenz
🔧
Technische Unterstützung

IT-Tandem-Partner (Modulbetreuer)

→ Technischer Ansprechpartner für den Fachbereich während der Testphase
→ Unterstützt bei der Fehleranalyse und -behebung
→ Klärt, ob ein Defect im Customizing, in der Entwicklung oder in der Basis gelöst werden muss
Kapitel 3 – Defect-Management

Der Prozess

1

Tester meldet Defect

Der Tester dokumentiert den Fehler direkt im Testfall. Ein Defect-Objekt wird angelegt mit: Beschreibung, Priorität, Screenshot/Evidenz, betroffener Testschritt.

↓
2

Key User/Koordination

Prüft: Ist der Fehler IT-relevant (Systemfehler, fehlendes Customizing) oder fachlich lösbar (falsche Testdaten, Bedienfehler)?

↓
3

IT-Klärung im Regeltermin

Wenn IT-relevant: Besprechung mit dem zugeordneten IT-Tandem-Partner. Der IT-Mitarbeiter klärt, ob er das Problem selbst lösen kann oder ob Entwicklung oder Basis involviert werden müssen.

↓
→ EntwicklungZuweisung an Entwicklungsteam
→ BasisZuweisung an Basis-Team
↓
5

Lösung und Nachtest

Fehler wird behoben und die Lösung im Defect dokumentiert. Status wechselt auf „Nachtest erforderlich“. Der ursprüngliche Tester führt einen Re-Test durch. Bei Erfolg: Defect wird geschlossen. Bei erneutem Fehler: Defect wird zurückgegeben.
